Bases de Données
Chapitre 1 : Conception de Bases de Données
1. Introduction
Une base de données est un ensemble organisé de données stockées de manière structurée pour permettre un accès, une gestion et une mise à jour efficaces. Avant de créer une base, il est crucial de bien la concevoir pour assurer la cohérence, l'efficacité et la fiabilité des données.
2. Étapes de conception d'une base de données
  • Analyse des besoins : Identifier les informations à stocker et les utilisateurs.
  • Modélisation conceptuelle : Modèle Entité-Association (E/A).
  • Modélisation logique : Traduire en modèle relationnel.
  • Modélisation physique : Définir les tables, types de données, index.
3. Modèle Entité-Association (E/A)
Le modèle E/A permet de décrire les entités, les attributs et les relations entre elles.
  • Entité : Objet identifiable (ex : Client, Produit).
  • Attribut : Propriété d'une entité (ex : nom, adresse).
  • Relation : Lien entre entités (ex : ACHETER entre Client et Produit).
Exemple : Un client passe une commande
Entités : CLIENT(id, nom, email)
COMMANDE(id, date)
Relation : PASSE(id_client, id_commande)
4. Modèle relationnel
Le modèle relationnel utilise des tables (relations) composées de lignes (tuples) et de colonnes (attributs).
  • Table : Représentation d'une entité ou d'une relation.
  • Clé primaire : Identifiant unique d'une ligne.
  • Clé étrangère : Référence à une clé primaire d'une autre table.
Exemple :
CLIENT(id PRIMARY KEY, nom, email)
COMMANDE(id PRIMARY KEY, date, id_client FOREIGN KEY REFERENCES CLIENT(id))

5. Normalisation
La normalisation vise à organiser les données pour réduire la redondance et éviter les anomalies. Elle se fait par niveaux appelés formes normales.
  • 1NF : Valeurs atomiques, pas de colonnes multiples.
  • 2NF : Dépendances fonctionnelles partielles éliminées.
  • 3NF : Pas de dépendances transitives.
Exemple : Client(id, nom, ville, code_postal)
-> ville et code_postal sont mieux dans une table VILLE(id, nom, code_postal)

6. Bonnes pratiques
  • Bien comprendre les besoins avant de modéliser
  • Utiliser des clés primaires claires et stables
  • Séparer les données répétitives dans des tables distinctes
  • Documenter le schéma conceptuel et logique
7. Résumé du chapitre
  • Concevoir une base commence par l'analyse des besoins
  • Le modèle E/A permet de structurer les entités et leurs relations
  • Le modèle relationnel permet de les transformer en tables
  • La normalisation évite la redondance et améliore la cohérence
